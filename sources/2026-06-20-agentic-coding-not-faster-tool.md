## Summary

Lengel-Zigich argues that the critical mistake engineering leaders will make is treating agentic coding as a faster version of existing workflows, rather than recognizing it as a fundamental constraint change requiring organizational redesign.

The core insight: **tool rollout is not the end of the work; it's the beginning.** When a core constraint shifts (human attention bottleneck → model capability bottleneck), practices built around the old constraint become misaligned. Many engineering practices—code review, planning, style guides, sprint rituals, test strategy, architecture—were optimized for human attention scarcity. Some are now "mispriced."

Three qualities will separate teams that actually improve from those that stall:

1. **Intellectual flexibility**: Treat practices as hypotheses, not identity. Ask what value each originally created and whether it still applies in agent-assisted contexts.

2. **Process curiosity**: When agents fail, dig into the system: missing context? Poor framing? Inadequate tests? Hard-to-navigate codebase? Review gaps? Task granularity? Fastest-improving teams treat agent development as a *work system*, not an individual skill.

3. **Evaluation discipline**: Tool selection cannot be based on preference or hype. Different models suit different work types (planning vs. implementation vs. refactoring vs. debugging). Teams need structured evaluation against real codebases, not generic benchmarks.

## Key quote

> "The mistake is thinking the rollout is done when every engineer has access to a coding agent. That is when the (new) management work starts."

## Takeaways

- Agentic coding reveals misalignment between old organizational practices and current constraints; leaders must actively question which practices are still fit-for-purpose
- Process diagnosis (not just blame) becomes a core management competency—studying failure modes systematically across the work system
- Tool evaluation requires empirical rigor and domain-specificity; generic benchmarks and vendor preference are insufficient
- Organizational redesign—not just tool distribution—determines whether teams transform or stall
- The shift is from managing individual developer skill to managing the entire agent-assisted workflow (framing, context, testing, review)
