## Summary

AI DevKit is a local-first operating layer that unifies multiple AI coding agents (Claude Code, Cursor, GitHub Copilot, Devin, etc.) under a single control plane. The core problem it solves: teams with multiple coding agents end up with scattered setup, no shared visibility, and agents that can't communicate or access shared context.

Key architectural components:

1. **Unified configuration** (`.ai-devkit.json`): Single source of truth reconciling setup across all agent tools
2. **Agent console**: TUI dashboard (`agent console`) for live supervision of running agents across providers
3. **Cross-agent communication** (`agent send`): Route prompts, logs, test output to running agents; supports `--stdin`, `--wait`, agent groups
4. **Memory retrieval without context bloat**: `@ai-devkit/memory` stores decisions, conventions, fixes in local SQLite; agents query when needed instead of carrying everything in every prompt
5. **Composable engineering skills**: `dev-lifecycle` (requirements → design → planning → implementation → testing → review) plus `verify`, `tdd`, `review`, `debugging`, `security`, `docs`, `simplification` skills

Installation is 30-second init (`npx ai-devkit@latest init`) with interactive wizard. Writes project-local `.ai-devkit.json`, agent-specific directories (`.claude/`, `.cursor/`, etc.) with skills and MCP wiring, and `docs/ai/` phase folders.

Operational commands include `agent list`, `agent detail`, `agent console`, `agent send "prompt" --id name --wait`, and stdin piping (`npm test 2>&1 | ai-devkit agent send --id name --stdin`).

## Problem framing (before/after)

Before: Multiple terminals, separate CLAUDE.md / .cursor/rules / AGENTS.md per tool, manual context/log copying, agents repeating project rules, no shared memory, agents diving into code without planning.

After: Shared setup, unified control surface, searchable memory, communication paths, reusable skills that travel with repo without bloating prompts, `dev-lifecycle` enforces requirements → design → planning → implementation → testing → review phase gates.

## Takeaways

- **Multi-agent coordination is operationalizable via unified config + communication layer**: Agents don't need centralized orchestration; local-first control plane with addressable agents solves the coordination problem without requiring agent-to-agent APIs.
- **Memory is retrievable-not-injected**: Storing conventions and decisions in queryable form (SQLite + MCP) avoids context bloat while ensuring agents can surface relevant history on demand.
- **Dev lifecycle (phases) is enforcer**: The workflow doesn't just recommend requirements → design → planning → implementation → testing → review; it's baked into skills and console, so agents can't skip phases.
- **Local-first with cross-provider reconciliation**: Single config works across Claude Code, Cursor, GitHub Copilot, Gemini, Devin because the init script normalizes their rule/setting files into a common structure.
- **Addressability enables async work**: `agent send --wait` and `--stdin` mean one human or agent can route work to running agents without copy-paste or context switching.
