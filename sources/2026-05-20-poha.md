# POHA (Personal Overnight Helper Agent)

> Caveats: README fetched via WebFetch (paraphrasing through a small model — quotes are reported, not guaranteed verbatim). The repo was **created and first pushed on 2026-05-18**, two days before this capture — 80 stars and 23 forks are early-viral numbers, not signal of operational maturity. Treat all claims about workflow durability as aspirational.

## Summary

POHA is a personal "ambient agent" that runs on a schedule overnight and delivers a morning briefing email before your alarm. Built on top of Claude Desktop's Cowork mode (with MCP connectors for Gmail, Google Calendar, WhatsApp). MIT-licensed. Author goes by `jigripokri` on GitHub; no bio.

The pitch is sharp: *"Most 'AI assistants' are reactive… That's a tool, not a brain extension."* POHA's bet is that the right shape for a personal agent is **observation + scheduled work + a few on-demand commands** — not a chat window you have to open.

The architecture splits into three layers:

1. **Briefing engine** — seven scheduled tasks, listed in `BACKLOG.md`. Key ones: 5am morning brief (scans WhatsApp, Gmail, Calendar; generates a brief with action-item links), 9pm reflection, Sunday 9am weekly review with 3-week memory pruning, daily 7am commitments-to-calendar, Friday noon "GSD" (Get Stuff Done) agenda + weekend events, monthly 28th financial audit.
2. **Memory system** — seven plain markdown files (`people`, `commitments`, `life`, `insights`, `finances`, `health`, `private carry-on`). Hand-editable, version-controllable, grep-able. The README states the philosophy bluntly: *"No vector DB. No RAG. No fine-tuning. Just files."*
3. **Skills** — five on-demand slash commands: `/draft`, `/reply`, `/wildcard`, `/gsd`, `/roast`.

A clever low-tech mechanism: the **acknowledgment loop**. The morning brief contains one-tap `mailto:` links; tapping one fires an email back to yourself with a coded subject like `EXO_DONE::action-slug`. The next scheduled run parses these emails and updates state. No webhook server, no UI — just IMAP and string parsing. This is the design concept the author wants credited.

Attention is filtered by a "tier-weighted" priority system (relationships ranked by importance, used to surface or suppress messages). Implementation details not in the README summary I retrieved.

Setup is honestly described as *"the configuration is the product"* — ~3 hours of bootstrapping with templated placeholders in `CLAUDE.md` (`{{USER_NAME}}`, `{{PARTNER_NAME}}`, tier definitions), MCP wiring, and a conversation with Claude to populate the memory files. Claimed ongoing cost: ~5 minutes per week to skim briefings. Privacy posture is reasonable: memory directory gitignored by default, "private carry-on" file gated and never auto-read, only Claude API calls and self-emails leave the device.

## Key quotes

> "Most 'AI assistants' are reactive… That's a tool, not a brain extension."

The framing thesis. POHA's design pivot is to invert the interaction model: the agent initiates, you skim.

> "No vector DB. No RAG. No fine-tuning. Just files."

The architectural manifesto. Read as a direct counter-position to systems like claude-mem (Postgres + Chroma + Redis + BullMQ). Worth taking seriously: for a single-user agent with bounded memory, a few hand-curated markdown files may genuinely outperform an embedding pipeline on signal-per-effort.

> "POHA runs in the background; you barely interact with it."

The success criterion. If you find yourself opening the chat window, the system isn't working.

> "The configuration is the product."

Honest acknowledgement that setup is not trivial. The customised `CLAUDE.md`, tier definitions, and memory files *are* what makes POHA POHA — strip them out and you have generic Claude.

## Takeaways

- **Scheduled "ambient" agents are a genuine alternative interaction model.** The default mental shape of an AI assistant is "I type, it responds." POHA is built around "it runs while I sleep, I read what it found." For tasks with periodic structure (mornings, weekends, month-end), the scheduled shape is much higher leverage than a chat session.
- **The "acknowledgement loop via mailto" pattern is reusable.** Want bidirectional state updates from a recipient without standing up a webhook server or app? Embed `mailto:` links with coded subjects and parse them on the next scheduled run. Low-tech, robust, almost free.
- **Files-as-memory is a credible architecture, not a fallback.** This repo and our own KB use the same shape (markdown + frontmatter, no vector store). For single-user agents with curated knowledge bounded to dozens or hundreds of files, the simpler approach is genuinely competitive with embedding pipelines on signal-per-effort — and dramatically easier to audit, edit, and version.
- **POHA and claude-mem are opposite ends of a real design spectrum.** Heavyweight auto-capture (claude-mem: hooks → queue → Postgres → Chroma → MCP retrieval) vs lightweight curated files (POHA: scheduled runs → markdown → grep). Same problem (persistent agent memory), genuinely different solutions. Worth a synthesis note once we have 1–2 more data points.
- **"The configuration is the product" deserves serious thought.** For agents that act on your behalf in messy personal-life domains (relationships, finances, health), the value is in the customisation, not the generic capability. Templates and placeholders are first-class.
- **Treat brand-new viral repos with discipline.** 80 stars in 48 hours is a Twitter spike, not a recommendation. Re-evaluate in 60 days: still maintained? users beyond the author? actual issues filed?
