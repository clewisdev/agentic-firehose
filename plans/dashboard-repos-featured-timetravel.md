# Plan: featured repo widget + time-travel repos panel

## Two related features

These are designed together because they share the same data model change (`stars_at_capture` frontmatter, see `plans/stars-at-capture-convention.md`) and the same conceptual frame: repos as a timeline, not a static list.

---

## Feature 1: Featured repo widget

A manually-curated "this week's pick" panel on the dashboard. Unlike the trending repos panel (ordered by stars, fully automated), this is an editorial choice — the owner highlights a specific repo with a short note on why it's worth attention right now.

### Data model

A new file `docs/featured.json` (committed manually, not generated by `build.py`):

```json
{
  "repo": "mvanhorn/last30days-skill",
  "note": "Real-time community signal across 7 platforms before calls. Reached 39k stars in 3 months. The multi-source synthesis pattern is directly applicable to agentic KB workflows.",
  "set_on": "2026-06-16"
}
```

`build.py` reads this file and includes it in `data.json`. The dashboard renders it as a distinct panel (top of right column, above recent activity, or as a banner).

### Dashboard panel design

- Repo name as a link to the GitHub URL
- Star count from the repos data
- Owner's note (2–3 sentences max)
- "Set on" date
- Visual distinction from the automated panels (different accent colour, maybe `--brass`)

### Update flow

Owner edits `docs/featured.json` directly (or asks the agent to update it). The next push triggers a redeploy.

## Feature 2: Time-travel repos panel

The trending repos panel currently shows a static list ordered by stars. Time-travel would let the reader step through months to see which repos were newly captured and what their stars were at that point.

### The problem with naive approaches

You cannot reconstruct "what did January look like" from today's data if you only store the current star count. A repo captured in January with 2k stars that now has 50k will look like it was always popular.

### Proposed data model: `stars_at_capture` frontmatter

See `plans/stars-at-capture-convention.md` for full detail. The short version: when a repo source is first captured, write `stars_at_capture: N` to its frontmatter. This field is immutable — it records the count at capture time, not today's count.

### Time-travel logic

`build.py` groups repo sources by `captured` month and emits a `repos_by_month` structure:

```json
{
  "repos_by_month": {
    "2026-05": [
      {"name": "safishamsi/graphify", "stars_at_capture": 55200, ...},
      {"name": "mattpocock/skills", "stars_at_capture": 96728, ...}
    ],
    "2026-06": [
      {"name": "mvanhorn/last30days-skill", "stars_at_capture": 39000, ...},
      {"name": "openclaw/openclaw", "stars_at_capture": 379000, ...}
    ]
  }
}
```

### Dashboard UX

The trending repos panel gains a month selector: `◂ May 2026 | Jun 2026 ▸`. The current month shows repos captured that month, sorted by `stars_at_capture`. Navigating backwards replays the KB's capture history.

The "All time" view (current behaviour) remains the default — shows all repos sorted by current star count.

### Edge cases

- A repo captured in multiple months (recaptured): show in the first month only, or show with updated stars in each month? First-capture-only is simpler and more honest.
- Repos captured before `stars_at_capture` was introduced: fall back to the corpus-scan count, displayed with a `~` prefix to indicate it's approximate.

### Relationship to featured repo

The featured repo could also show historical context: "when this was featured in June 2026, it had 39k stars."

---

## Implementation sequence

1. Land `stars-at-capture-convention.md` and update the Worker to write the field
2. Backfill `stars_at_capture` on existing repo sources where the count is known
3. Implement `featured.json` + dashboard panel (standalone, no dependencies)
4. Add `repos_by_month` to `build.py` output
5. Implement time-travel UX in the dashboard

## Open questions

- What is the right granularity for time travel — monthly or weekly? Monthly is probably right given capture volume.
- Should the featured panel be on the main dashboard view or a separate page?
- Is `docs/featured.json` the right location, or should it live at the repo root?
